GitHub ActionsのCI（ワークフロー）の失敗ログを取得・分析し、バグの原因を日本語で提示した後、コードを修正する。

## 前提

- `gh` CLI が利用可能で、認証済みであること（`gh auth status` で確認できる）。
- カレントディレクトリがGitHubリポジトリであること。

## 手順

### 1. 失敗したCI実行を特定する

最近のワークフロー実行を確認し、失敗したものを探す:
```
gh run list --limit 10
```
- ユーザーが特定の実行（run）やブランチ・PRを指定している場合は、それを対象にする。
- 失敗（`failure`）した実行が複数ある場合は、対象をユーザーに確認する。
- 失敗が見つからない場合は、その旨を伝えて終了する。

### 2. 失敗ログを取得する

対象の実行IDを使って、失敗したジョブ・ステップのログを取得する:
```
gh run view <run-id> --log-failed
```
- 必要に応じて `gh run view <run-id>` でジョブ構成全体を確認する。
- ログが長い場合は、エラーメッセージ・スタックトレース・失敗したステップに絞って読む。

### 3. 失敗の原因を分析する

- ログからエラーの種類を特定する（テスト失敗 / ビルドエラー / Lintエラー / 型エラー / 依存関係 / 環境設定など）。
- エラーに関連するソースコード・設定ファイルを読み、根本原因を突き止める。
- ワークフロー定義（`.github/workflows/*.yml`）に問題がある場合もあるので、必要なら確認する。

### 4. 原因をユーザーに提示する

修正に着手する前に、以下を日本語で説明する:
- **どのジョブ/ステップが失敗したか**
- **エラーの内容**（該当ログを引用）
- **根本原因**（なぜ失敗したのか）
- **修正方針**（どう直すか）

### 5. 修正する

- 特定した原因に基づいてコードまたは設定を修正する。
- 修正後、可能であればローカルで該当のテスト・ビルド・Lintを実行して検証する。
- 何を変更したか、なぜその変更で直るのかを日本語で説明する。

## 注意

- 説明・提示はすべて日本語で行う。
- 原因が複数ある、または修正方針が複数考えられる場合は、選択肢を提示してユーザーに確認する。
- 推測で修正せず、必ずログとソースコードで原因を裏付けてから修正する。
- ログだけで原因が断定できない場合は、不足している情報を明示する。
- コミットやpushはユーザーから明示的に指示があった場合のみ行う。

---

## 使い方

Claude Code上で以下を入力するだけで実行されます。
/fix-ci

特定の実行を対象にしたい場合は、run IDやPR番号を添えて指示してください。
例: 「/fix-ci の PR #42 の失敗を直して」
