自分にアサインされているGitHub issueを一覧で取得し、どのissueを対象にするかユーザーに確認する。選ばれたissueの概要を参照し、リポジトリのコードやWeb（公式ドキュメント等）で**調べられる範囲**で調査を行い、調査でわかった詳しい内容をissueにコメントとして追記する。**追記は長くしすぎず、要点を簡潔にまとめる。**

## 前提

- `gh` CLI が利用可能で、認証済みであること。
- GitHubリポジトリ内であること。

## 手順

### 1. アサインされているissueを取得する（許可不要）

自分にアサインされているissueをすべて取得する:
```
gh issue list --assignee "@me" --state open --json number,title,labels,updatedAt
```
- 必要に応じて `--state all` も検討するが、基本はオープンなものを対象にする。
- 1件も無い場合は、その旨を伝えて終了する。

### 2. 対象issueをユーザーに確認する

- 取得したissueを番号・タイトル付きの一覧で提示する。
- **どのissueを調査・追記対象にするかユーザーに尋ねる。** ユーザーが選ぶまで先に進まない。
- ユーザーが最初から特定のissue番号を指定している場合は、それを対象にしてよい。

### 3. issueの概要を読み取る（許可不要）

選ばれたissueの本文・既存コメントを取得する:
```
gh issue view <number> --comments
```
- 概要、目的、再現手順、要望などを把握する。
- 何を調査すべきか（不明点・確認すべき仕様・関連実装など）を整理する。

### 4. 調べられる範囲で調査する（許可不要）

issueの内容に応じて、可能な手段で調査する:
- **コード調査**: issueに関連するファイル・関数・設定をリポジトリ内で検索して読む（該当箇所・現状の実装・影響範囲など）。
- **Web調査**: 外部ライブラリ・API・エラー・仕様に関わる場合、`WebSearch` / `WebFetch` で公式ドキュメント等を調べる。
- 調査で確認できた事実のみを根拠にする。推測は推測と明示する。
- **調べられない／情報が足りない部分は、その旨を明記する**（無理に断定しない）。

### 5. 追記内容をまとめる（簡潔に）

調査結果を、issueへの追記コメントとして簡潔にまとめる。長文化を避け、要点に絞る:
- **わかったこと**（調査で判明した事実。関連コードは `path:line` で示す）。
- **想定される対応方針／論点**（あれば、箇条書きで簡潔に）。
- **未確認・要確認の点**（調べきれなかったこと）。
- 参照したコード箇所・URLは出典として簡潔に添える。

長さの目安: 冗長な前置きや繰り返しは入れず、読み手がすぐ把握できる分量に収める。

### 6. 追記内容をユーザーに提示して確認する

- まとめたコメント案を提示する。
- **このままissueに追記してよいかユーザーに確認する。** 承認されるまで投稿しない。
- 修正指示があれば調整して再提示する。

### 7. issueに追記する（承認後）

承認されたら、issueにコメントを追加する:
```
gh issue comment <number> --body "<まとめたコメント>"
```
- 投稿後、対象issueのURLを報告する。

## 注意

- 説明・追記内容はすべて日本語で行う。
- **追記は簡潔に。** 情報を盛り込みすぎず、要点に絞る。
- 調査で裏付けが取れた内容と、推測・未確認の内容を明確に区別する。
- issueへの投稿は必ずユーザーの承認を得てから行う。
- Web調査では最新かつ公式の情報を優先する。

---

## 使い方

Claude Code上で以下を入力するだけで実行されます。
/investigate-issue

特定のissueを最初から指定することもできます。
例: 「/investigate-issue #42 を調査して」
