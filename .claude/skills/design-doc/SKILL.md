新機能の設計をするためのSkill。ユーザーへのヒアリング（プロンプト入力工程）を挟んで要件を固め、Mermaid図を含むMarkdownの設計ドキュメントを作成する。出力先は既存ドキュメントの配置に倣って決める。新機能追加に伴って修正が必要な既存ドキュメント（全体ER図など）も更新する。作成後は、関連技術の最新の公式ドキュメントを調査し、設計の妥当性を改めて検証する。

## 全体方針（重要）

- **所要時間**: すべての処理を含めて**10分以上**かけ、丁寧に作成する。調査・検証を省略して急がない。
  - ただし**全工程で20分を超える場合は処理を中断**し、その時点までの成果と残作業をユーザーに報告して指示を仰ぐ。
- **許可ポリシー**: 以下のタイミングで必ずユーザーに許可を求める。
  1. **新規ドキュメントを出力（ファイル作成）する前**。
  2. **既存ドキュメントを修正する前**（対象ファイルごと、または一括で対象を提示して許可を得る）。
  - それ以外（ヒアリング、既存構成の調査、Web調査、設計案の検討）は許可を求めず進めてよい。
- すべての出力・説明・ドキュメント本文は日本語で作成する。

## 手順

### ステップ0: 開始時刻の記録

- 処理開始時に現在時刻を確認しておく（`date` 等）。経過時間を意識し、10分以上・20分以内に収める判断材料にする。

### ステップ1: 要件ヒアリング（プロンプト入力工程）

新機能について、ユーザーに以下を質問して要件を固める。一度に全部ではなく、不足している点を中心に対話的に確認する:
- 新機能の概要・目的（何を解決するか）。
- 主なユースケース／ユーザーストーリー。
- 関連するデータ（新規・変更されるエンティティ、項目）。
- 既存システムとの連携・依存関係。
- 想定している技術スタック・ライブラリ（あれば）。
- 非機能要件（性能・セキュリティ・スケール等、あれば）。

要件が曖昧な場合は、設計に入る前に必ず確認する。

### ステップ2: 既存ドキュメント構成の調査（許可不要）

- リポジトリ内のドキュメント配置を調査する（`docs/`, `documentation/`, `design/`, `*.md` など）。
- 既存ドキュメントの命名規則・ディレクトリ構成・記法（Mermaidの使われ方など）を把握する。
- **出力先の候補**を決める: 既存の設計ドキュメントが置かれている場所に倣う。該当が無ければ `docs/design/` などを提案する。
- 新機能に伴い**修正が必要そうな既存ドキュメント**を洗い出す（例: 全体ER図、アーキテクチャ図、API一覧、用語集など）。

### ステップ3: 設計ドキュメントの作成（メモリ上で組み立て）

以下を含むMarkdownの設計ドキュメントを組み立てる。機能に応じて適切な図を選ぶ:
- **概要 / 目的 / 背景**
- **要件**（機能要件・非機能要件）
- **データモデル**: Mermaidの **ER図**（`erDiagram`）
- **処理の流れ**: Mermaidの **フロー図**（`flowchart`）や **シーケンス図**（`sequenceDiagram`）
- **アーキテクチャ / コンポーネント構成**（必要に応じて）
- **API / インターフェース設計**（必要に応じて）
- **既存システムへの影響**（修正が必要な箇所の一覧）
- **考慮事項・代替案・未決事項**

Mermaidはコードフェンス（```mermaid）で記述する。既存ドキュメントの記法に合わせる。

### ステップ4: 関連技術の最新公式ドキュメント調査（許可不要）

設計案ができた後、**改めてその設計が適切かを検証する**ために、関連技術の最新の公式ドキュメントなどを調査する:
- 採用予定のフレームワーク・ライブラリ・サービスについて、`WebSearch` / `WebFetch` で**最新の公式ドキュメント**を調べる（バージョン・推奨パターン・非推奨API・ベストプラクティス）。
- Claude/Anthropic関連技術が含まれる場合は claude-api Skill も参照する。
- 調査結果を設計に反映する（古いパターンを使っていないか、より適切な方法が無いか）。
- 調査で参照したURLはドキュメントの「参考資料」セクションに出典として残す。

### ステップ5: 新規ドキュメントの出力　→【許可ポイント①】

- 出力先パスと、作成するドキュメントの内容（要約＋主要なMermaid図）をユーザーに提示する。
- **ここでファイル作成の許可を求める。** 承認されてから書き込む。

### ステップ6: 既存ドキュメントの修正　→【許可ポイント②】

- ステップ2で洗い出した、修正が必要な既存ドキュメントについて、対象ファイルと変更内容（差分の要約、更新後のER図など）を提示する。
- **ここで修正の許可を求める。** 承認されてから修正する。
- 修正不要と判断した場合はその旨を説明する。

### ステップ7: 完了報告

- 作成・修正したファイルの一覧、設計の要点、調査で確認した最新情報のポイント、未決事項を日本語で報告する。
- かかった時間の目安も添える。

## 注意

- 10分未満で終わりそうな場合は、調査・代替案検討・図の充実を深め、品質を高める方向に時間を使う（拙速にしない）。
- 20分を超えそうな場合は中断し、途中成果（下書き）と残作業を提示して指示を仰ぐ。
- ファイルの新規作成・既存修正は、必ず該当する許可ポイントで承認を得てから行う。
- Web調査では「最新」かつ「公式」の情報を優先する。バージョン差異に注意し、参照日を明記する。
- 推測で設計を断定せず、不明点はヒアリングまたは調査で裏付ける。
