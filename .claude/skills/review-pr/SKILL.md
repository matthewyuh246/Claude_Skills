現在いるブランチに紐づくGitHub PRの変更ファイルをレビューし、**確実に修正した方がよい箇所のみ**を最小限に絞って指摘し、GitHubのPRにコメントを投稿する。

## 前提

- `gh` CLI が利用可能で、認証済みであること（`gh auth status` で確認できる）。
- 現在のブランチにPRが存在すること。

## レビュー方針（重要）

- **最小限の指摘に徹する。** 数を出すことが目的ではない。
- 指摘するのは「**確実に修正した方がよい**」もののみに限定する:
  - バグ・ロジックの誤り（明確に動作がおかしくなるもの）
  - Nullや例外で落ちる箇所、境界条件の漏れ
  - セキュリティ上の問題（機密情報の混入、インジェクション等）
  - リソースリーク、明らかな性能上の致命的問題
  - 明確な誤り（typo含む）で動作・意味に影響するもの
- **指摘しない**もの:
  - 好みレベルのスタイル・命名（動作に影響しないもの）
  - 「こうもできる」程度の代替案
  - 確信が持てない推測ベースの指摘
- 確実に修正すべき箇所が無ければ、無理に指摘を作らず「重大な問題は見つからなかった」とコメントする。

## 手順

### 1. 対象PRを特定する

現在のブランチのPRを確認する:
```
gh pr view --json number,title,headRefName,baseRefName
```
- PRが見つからない場合は、その旨を伝えて終了する。

### 2. 変更ファイルと差分を取得する

```
gh pr diff
```
- 変更されたファイル一覧と差分を読み取る。
- 文脈が必要な場合は、該当ファイルをローカルで開いて前後を確認する。

### 3. 差分をレビューする

- 上記「レビュー方針」に従い、確実に修正すべき箇所のみを抽出する。
- 各指摘について、ファイル名・行・問題内容・修正案を整理する。
- 指摘は必ず差分（追加・変更された行）に基づいて行う。

### 4. 指摘内容をユーザーに提示する

コメント投稿の前に、以下を日本語でまとめて提示する:
- 指摘箇所ごとに: **ファイル:行 / 問題 / なぜ直すべきか / 修正案**
- 指摘が無い場合はその旨を伝える。
- この内容でコメント投稿してよいかユーザーに確認する。

### 5. GitHubのPRにコメントする（承認後）

承認されたら、以下のいずれかでコメントする。

- **特定の行へのインラインコメント**（推奨。複数指摘をまとめて1レビューにする）:
  `gh api` で Pull Request review を作成する。
  ```
  gh api repos/{owner}/{repo}/pulls/{pr-number}/reviews \
    -f event=COMMENT \
    -f body="<レビュー全体の概要>" \
    -f 'comments[][path]=<ファイルパス>' \
    -f 'comments[][line]=<行番号>' \
    -f 'comments[][side]=RIGHT' \
    -f 'comments[][body]=<指摘内容と修正案>'
  ```
  - `{owner}/{repo}` は `gh repo view --json owner,name` 等で取得する。
  - 行番号は差分の新しい側（RIGHT）の行を指定する。
  - 複数指摘がある場合は `comments[]` を繰り返して1回のレビューにまとめる。

- 行指定が難しい・全体への総評のみの場合は、本文コメントを使う:
  ```
  gh pr review --comment -b "<コメント本文>"
  ```

- 投稿後、コメントしたPRのURLを報告する。

## 注意

- 指摘・コメントはすべて日本語で行う。
- 推測で指摘しない。確信が持てるもののみコメントする。
- コメントは `--comment`（レビュー意見）で行い、`--approve` や `--request-changes` はユーザーの明示的な指示がない限り使わない。
- 投稿は必ずユーザーの承認を得てから行う。
